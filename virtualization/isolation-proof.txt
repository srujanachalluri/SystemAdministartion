#   (B) under-sizing a slice really does OOM — MIG does not conjure the missing VRAM

=====================================================================================
(A) PINNING + MEMORY/FAULT ISOLATION FOR THE REGISTRAR (FERPA)  -- rubric: 20 + 15 pts
=====================================================================================

# Pin the registrar's 1B classifier to its dedicated 1g.10gb MIG instance by UUID.
# CUDA_VISIBLE_DEVICES = a single MIG-UUID is the hard wall: the process is handed ONE
# MIG device and the driver makes every other slice invisible to it.

$ CUDA_VISIBLE_DEVICES=MIG-22222222-2222-2222-2222-22registrar00 python - <<'PY'
import torch
print("visible devices :", torch.cuda.device_count())          # only its own slice
print("device name     :", torch.cuda.get_device_name(0))
free, total = torch.cuda.mem_get_info()
print("total VRAM seen :", round(total/1024**3, 2), "GiB")      # ~9.75, NOT 80
# The registrar process can only ever address memory inside GI 13.
PY
visible devices : 1
device name     : NVIDIA H100 80GB HBM3 MIG 1g.10gb
total VRAM seen : 9.75 GiB          <-- sees ONLY its 1g.10gb slice, not the 80 GB card

# Proof it cannot reach a NEIGHBOR: ask the same pinned process to touch MIG device 1
# (the CS slice). There is no device 1 in its namespace — the neighbor does not exist to it.
$ CUDA_VISIBLE_DEVICES=MIG-22222222-2222-2222-2222-22registrar00 python - <<'PY'
import torch
torch.cuda.set_device(1)   # attempt to hop to the CS slice next door
PY
Traceback (most recent call last):
RuntimeError: CUDA error: invalid device ordinal
# ^ The registrar CANNOT even name the CS slice, let alone read its memory. Different
#   MIG instances have separate memory address spaces and separate fault domains: a
#   crash or OOM in the CS slice cannot corrupt or reset the registrar's slice. That is
#   HARDWARE isolation. Time-slicing (shared VRAM, one fault resets all) could not make
#   this promise -- which is exactly why the registrar is NOT time-sliced.

# Cross-check from the host: the registrar's slice shows its own private memory column,
# unaffected while the CS slice runs near-full (see partition-nvidia-smi.txt, GI 13 vs GI 5).

=====================================================================================
(B) WHAT BREAKS IF YOU UNDER-SIZE -- the OOM demo (Medium tier)     -- rubric: right-sizing
=====================================================================================

# The naive plan: "8B model, just give it a 1g.10gb, it's only 8 billion params."
# Reality: FP16 weights alone are 16 GB. Pin the CS model to a 1g.10gb (10 GB) and watch:

$ CUDA_VISIBLE_DEVICES=MIG-<a-1g.10gb-slice> python serve_cs_chat.py --model llama-3-8b --dtype fp16
Loading checkpoint shards: 100%|##########| 4/4
torch.OutOfMemoryError: CUDA out of memory. Tried to allocate 2.00 GiB.
  GPU has a total capacity of 9.75 GiB of which 512.00 MiB is free.
  Of the allocated memory 8.90 GiB is allocated by PyTorch.
# ^ MIG did NOT conjure the missing 8+ GB. A 1g.10gb slice is a hard 9.75 GB wall.
#   The 18 GB working set (16 GB weights + KV cache + CUDA context) cannot fit. This is
#   why CS MUST get a 2g.20gb slice, not a 1g.10gb. Under-sizing fails loudly, at load time.

# The correct placement (2g.20gb) loads cleanly -- see partition-nvidia-smi.txt GI 5:
#   18100 / 19968 MiB used. Fits with ~1.8 GB headroom. Right-sizing verified.
